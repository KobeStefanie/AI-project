## 二、实施内容

### 1. 创建了 `process_updates.py` 处理脚本

**文件位置**：`src/process_updates.py`

**功能**：
- 读取 `data/pending_updates.json` 中的待处理更新
- 支持三种更新类型：
  - `supervision_record_add`：添加感悟
  - `supervision_record_edit`：编辑感悟
  - `supervision_record_delete`：删除感悟
- 自动生成唯一的 record_id（格式：`sr_001`, `sr_002`...）
- 更新后自动调用 `generate_case_library.py` 重新生成HTML
- 完成后清空 `pending_updates.json`

**使用方法**：
```bash
python src/process_updates.py
```

### 4. 创建了 `pending_updates.json` 文件

**文件位置**：`data/pending_updates.json`

**作用**：
- 存储前端提交的更新请求
- 由 JavaScript 写入，由 Python 脚本读取
- 处理完成后自动清空

**数据格式示例**：
```json
[
  {
    "type": "supervision_record_add",
    "case_id": "C20260620001",
    "content": "第一次阅读这个案例，感觉...",
    "approach": "大观学派"
  },
  {
    "type": "supervision_record_edit",
    "case_id": "C20260620001",
    "record_id": "sr_001",
    "content": "重新理解后，我认为...",
    "approach": null
  },
  {
    "type": "supervision_record_delete",
    "case_id": "C20260620001",
    "record_id": "sr_002"
  }
]
```

## 三、数据结构

### supervision_records 字段

在 v2.0 案例JSON中，`supervision_records` 字段存储所有感悟记录：

```json
{
  "case_id": "C20260620001",
  "version": "2.0",
  "basic_info": {...},
  "analyses": {...},
  "supervision_records": [
    {
      "record_id": "sr_001",
      "content": "这个案例让我看到...",
      "approach": "大观学派",
      "created_at": "2026-06-28T12:30:00",
      "updated_at": "2026-06-28T12:30:00"
    },
    {
      "record_id": "sr_002",
      "content": "从CBT角度看...",
      "approach": "CBT",
      "created_at": "2026-06-28T14:00:00",
      "updated_at": "2026-06-28T15:20:00"
    }
  ]
}
```

**字段说明**：
- `record_id`：唯一标识符，格式 `sr_XXX`
- `content`：感悟内容（必填）
- `approach`：流派标注（可选，null表示不标注）
- `created_at`：创建时间（ISO 8601格式）
- `updated_at`：最后修改时间（ISO 8601格式）

## 四、完整工作流程

### 用户操作流程

1. **添加感悟**：
   - 打开案例详情页
   - 点击"添加感悟"按钮
   - 选择流派视角（可选）
   - 输入感悟内容
   - 点击"保存"
   - 页面显示提示：请运行 `python src/process_updates.py`

2. **编辑感悟**：
   - 点击感悟卡片上的"编辑"按钮
   - 修改流派或内容
   - 点击"保存"
   - 收到同样的提示

3. **删除感悟**：
   - 点击感悟卡片上的"删除"按钮
   - 确认删除
   - 收到同样的提示

4. **处理更新**：
   - 在终端运行：`python src/process_updates.py`
   - 脚本自动：
     - 读取所有待处理更新
     - 更新案例JSON文件
     - 重新生成HTML
     - 清空 pending_updates.json
   - 刷新浏览器查看更新

## 五、技术特点

### 1. 手动触发机制

**为什么采用手动方式**：
- 实现简单，无需后台服务
- 用户有明确的控制感
- 编辑案例不是高频操作
- 可以批量处理多个更新

**未来改进方向**：
- 如果体验不好，可以升级为文件监听自动处理
- 或者实现WebSocket实时更新

### 2. 数据安全

- 每次保存案例前会自动备份旧版本
- 备份位置：`data/backup/<case_id>/<timestamp>.json`
- 更新失败不会清空 pending_updates.json

### 3. 时间记录

- `created_at`：感悟首次创建时间，永不改变
- `updated_at`：感悟最后修改时间，每次编辑更新
- 界面显示：如果两者不同，显示"已编辑"标记

### 4. 流派标注可选

- 用户可以选择流派视角
- 也可以选择"不标注流派"
- `approach` 字段可以为 `null`
- 符合实际使用场景：有些感悟确实是跨流派的

## 十一、注意事项

### 1. 内容转义

JavaScript代码中对感悟内容进行了转义处理，防止：
- 单引号、双引号破坏HTML结构
- 换行符导致JavaScript语法错误
- 反斜杠转义问题

### 2. 服务器依赖

历次感悟功能依赖以下服务器：
- **case_api.py (5001)**：提供 pending_updates API
- 其他服务器不影响此功能

### 3. 并发安全

当前实现是简单的文件读写，不支持并发：
- 不要同时在多个浏览器窗口编辑
- 先处理完一批更新再提交下一批
- 如需并发支持，需要引入锁机制

### 4. 数据备份

虽然有自动备份，但建议：
- 定期手动备份整个 `data/` 目录
- 重要修改前先备份
- 备份位置：`data/backup/<case_id>/`
