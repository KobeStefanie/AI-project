## 一、系统定位

### 1.1 核心定位

这是一个**多流派心理咨询案例学习与督导系统**，而不仅仅是案例展示系统。

**核心特点**：
- 案例是中性的，不属于任何特定流派
- 同一个案例可以用多种心理学流派进行分析
- 随着学习深入，可以对案例进行迭代式的多角度理解
- 支持督导反思、案例编辑、模拟对话等学习功能

### 1.2 系统目标

1. **案例归档**：Word文档 → 结构化JSON + 多流派分析
2. **多维分析**：同一案例支持大观、CBT、精神动力学、人本、存在主义等多种视角
3. **学习迭代**：记录对案例的历次感悟，支持流派标注或纯感悟
4. **内容完善**：发现案例信息遗漏时可以直接编辑补充
5. **动态管理**：标签、危机评估、关键词等可以修改调整
6. **未来扩展**：预留模拟对话功能接口

## 二、核心需求

### 2.1 多流派分析架构

#### 需求描述
- 案例本身是中性的，不绑定任何流派
- 同一个案例可以有多个流派的分析视角
- 每个流派有自己的分析框架、概念体系、技术清单

#### 使用场景
**场景1：学习新流派后重新分析案例**
```
2026年6月：学习大观学派 → 用危机干预视角分析案例A
2026年8月：学习CBT → 用认知行为视角重新分析案例A
2026年10月：学习精神动力学 → 用潜意识视角再次分析案例A
```

**场景2：对比不同流派对同一案例的理解**
```
打开案例A详情页
- [大观学派] Tab：危机评估L级，使用情绪承接技术
- [CBT] Tab：识别灾难化思维，建议认知重构
- [精神动力学] Tab：发现防御机制，分析客体关系
- [对比视图]：看到不同流派的互补与冲突
```

#### 数据结构要求
```json
{
  "case_id": "C20260620001",
  
  // 基础层：流派无关
  "basic_info": {...},
  "dialogue": "...",
  
  // 分析层：每个流派独立
  "analyses": {
    "daguanpai": {
      "crisis_level": "L",
      "tags": {...},
      "techniques": [...]
    },
    "cbt": {
      "cognitive_patterns": [...],
      "intervention_plan": {...}
    },
    "psychodynamic": {...},
    "humanistic": {...},
    "existential": {...}
  }
}
```

#### 界面要求
- 案例详情页使用Tab切换不同流派视图
- 首页支持"按流派筛选"（查看哪些案例有CBT分析）
- 每个流派视图独立展示该流派的分析内容

### 2.2 历次感悟迭代

#### 需求描述
- 对同一案例多次阅读会有不同的感悟和发现
- 需要记录每次感悟的内容和时间
- 感悟可以标注流派视角（如"从CBT视角看..."），也可以是纯粹的个人感悟
- 支持对感悟进行修改和删除

#### 使用场景
**场景1：流派视角感悟**
```
第1次（2026-06-27，大观学派视角）：
"咨询师对幻听的隐喻技术很创新，但父母情感忽视问题没有深入探讨"

第2次（2026-07-20，CBT视角）：
"学习CBT后重新看，发现来访者有明显的灾难化思维和黑白思维"

第3次（2026-08-05，精神动力学视角）：
"从精神动力学看，进食障碍可能是对童年控制的反抗"
```

**场景2：纯粹的感悟（不标注流派）**
```
第4次（2026-09-10，无流派标注）：
"今天重读这个案例，突然理解了来访者说的'凌迟'，
那种痛苦不是某个流派能完全解释的，是存在性的"
```

**场景3：修改之前的感悟**
```
回到第1次感悟，点击"编辑"：
"修正：咨询师其实有触及家庭动力，我第一次看漏了"
系统记录：created_at: 2026-06-27, updated_at: 2026-09-15
```

#### 功能要求
1. **添加感悟**
   - 弹出表单
   - 可选：流派标注（下拉选择：大观/CBT/精分/人本/存在/无）
   - 必填：感悟内容（多行文本）
   - 自动记录创建时间

2. **编辑感悟**
   - 可以修改流派标注
   - 可以修改感悟内容
   - 自动更新"最后修改时间"

3. **删除感悟**
   - 确认后删除
   - 不可恢复（或需要从备份恢复）

4. **时间记录**
   - created_at：创建时间
   - updated_at：最后修改时间
   - 显示格式：2026-06-27 14:30

#### 数据结构
```json
{
  "supervision_records": [
    {
      "id": "sr_20260627143000",
      "created_at": "2026-06-27T14:30:00",
      "updated_at": "2026-06-27T14:30:00",
      "approach": "大观学派",  // 可选，可为null表示纯感悟
      "content": "咨询师对幻听的隐喻技术很创新..."
    },
    {
      "id": "sr_20260720100000",
      "created_at": "2026-07-20T10:00:00",
      "updated_at": "2026-07-20T10:00:00",
      "approach": "CBT",
      "content": "学习CBT后重新看..."
    },
    {
      "id": "sr_20260910160000",
      "created_at": "2026-09-10T16:00:00",
      "updated_at": "2026-09-10T16:00:00",
      "approach": null,  // 纯感悟，不标注流派
      "content": "今天重读这个案例，突然理解..."
    }
  ]
}
```

### 2.3 案例标签管理

#### 需求描述
- 看到案例库中某个案例的标签分配不对，可以直接在案例详情页修改
- 可以删除不准确的标签
- 可以添加缺失的标签
- 标签管理在流派视图内进行（如在"大观学派"Tab修改大观的标签）

#### 使用场景
```
打开案例详情页 → 切换到"大观学派"Tab
看到关系标签中有"社交关系-同伴关系-融入困难"
但我觉得应该是"社交关系-同伴关系-社交孤立"

操作：
1. 点击"编辑标签"按钮
2. 点击"融入困难"后面的×删除
3. 点击"+ 添加关系标签"
4. 从标签库选择"社交关系-同伴关系-社交孤立"
5. 点击"保存"
6. 系统更新JSON，重新生成HTML
```

#### 功能要求
1. **编辑模式切换**
   - 默认：只读展示
   - 点击"编辑标签"进入编辑模式
   - 每个标签后显示×删除按钮
   - 显示"+ 添加标签"按钮

2. **标签选择器**
   - 展示标签库的树形结构
   - 可搜索标签
   - 选中后添加到案例

3. **保存机制**
   - 点击"保存"后写入pending_updates.json
   - 运行Python脚本更新JSON
   - 自动更新索引
   - 重新生成HTML

### 2.4 案例内容编辑

#### 需求描述
- 发现案例本身有信息遗漏或错误时，可以直接编辑
- 可编辑的内容包括：基本信息、接访信息、危机评估、关键词、使用技术、AI分析、对话记录等
- 编辑是对案例本身的完善，成为案例的一部分

#### 使用场景
**场景1：补充遗漏信息**
```
发现案例基本信息中缺少"婚姻状况"
→ 点击"编辑基本信息"
→ 添加"婚姻状况：未婚"
→ 保存
```

**场景2：修正错误**
```
发现对话记录中有错别字
→ 点击"编辑对话记录"
→ 修正错别字
→ 保存
```

**场景3：重新评估危机等级**
```
重新阅读案例，觉得危机等级判断有误
→ 在大观学派Tab中点击"编辑危机评估"
→ 修改等级从M改为L
→ 补充证据
→ 保存
```

#### 功能要求
1. **字段级编辑**
   - 每个可编辑字段旁边有"编辑"按钮
   - 点击后弹出编辑表单
   - 修改后保存

2. **自动备份**
   - 编辑前自动备份原JSON到backup目录
   - 按日期组织：backup/20260627/C20260620001.json

3. **修改记录**
   - 记录最后修改时间（last_modified）

### 3.3 前后端交互机制

#### 编辑流程
```
1. 用户在HTML编辑
2. JavaScript收集数据
3. 写入 data/pending_updates.json
4. 页面提示"请运行：python src/process_updates.py"
5. 用户手动运行Python脚本
6. Python脚本：
   - 读取pending_updates.json
   - 更新案例JSON
   - 更新索引
   - 重新生成HTML
   - 清空pending_updates.json
7. 用户刷新浏览器，看到更新
```
