# 架构分析与改进方案

## 一、现有架构深度分析

### 1.1 当前系统定位

这不是一个简单的"案例展示系统"，而是一个**督导学习与成长系统**：

- **输入层**：原始接访记录（Word）
- **处理层**：AI辅助分析 + 人工审核
- **存储层**：结构化JSON + 索引系统
- **展示层**：HTML案例库（静态）
- **学习层**：⚠️ **缺失** - 督导反思、迭代学习、模拟练习

**核心问题**：当前架构是**单向数据流**（Word → JSON → HTML），缺少**反馈回路**。

### 1.2 现有数据流

```
Word文档（不可变）
    ↓ [AI处理]
JSON案例（结构化）
    ↓ [生成HTML]
HTML案例库（静态展示）
    ↓ [学习反馈] ← ⚠️ 这里断了
  ？？？
```

**断点**：用户在HTML看案例、产生感悟、发现错误，但无法回流到系统。

### 1.3 数据结构分析

**案例JSON的层次**：
```json
{
  "case_id": "C20260620001",              // 不可变（唯一标识）
  "source_file": "...",                   // 不可变（溯源）
  "created_at": "...",                    // 不可变（时间戳）
  
  "basic_info": {...},                    // 可编辑（补充遗漏）
  "session_info": {...},                  // 可编辑（修正错误）
  "tags": {...},                          // 可编辑（调整分类）
  "crisis_level": "L",                    // 可编辑（重新评估）
  "crisis_evidence": "...",               // 可编辑（补充证据）
  "keywords": [...],                      // 可编辑（增删关键词）
  "techniques_used": [...],               // 可编辑（补充技术）
  "dialogue": "...",                      // 可编辑（修正错别字）
  "ai_analysis": {...},                   // 可编辑（人工修正AI分析）
  
  "supervision_records": []               // 学习层（新增数据）
}
```

**分层原则**：
- **不可变层**：case_id、source_file、created_at（身份+溯源）
- **基础层**：AI自动提取的案例数据（可编辑完善）
- **学习层**：用户主动添加的反思数据（新增）

### 1.4 现有目录结构的问题

```
data/
├── cases/
│   ├── original/          # Word原件（不可变）✅
│   ├── processed/         # JSON案例（应可编辑，但当前无机制）⚠️
│   └── supervision/       # 督导记录（空目录，未使用）⚠️
├── index/                 # 4个索引文件（依赖processed/）⚠️
└── config/
    └── tags_library.json  # 统一标签库 ✅
```

**问题1**：`supervision/` 目录设计为独立markdown文件，但与案例JSON的 `supervision_records` 字段冲突。

**问题2**：索引文件与案例JSON的一致性如何保证？修改案例后索引不会自动更新。

**问题3**：没有编辑历史/版本控制机制。

## 二、用户需求的本质

### 2.1 三个需求的共性

| 需求 | 表面操作 | 本质需求 |
|------|---------|---------|
| 历次感悟 | 添加/编辑/删除感悟 | **反思迭代** - 在案例上叠加学习层 |
| 标签管理 | 修改案例标签 | **分类纠错** - 完善基础数据质量 |
| 案例编辑 | 修改案例各字段 | **数据完善** - 补充遗漏和修正错误 |

**共同点**：都是**在静态数据上添加动态编辑能力**。

### 2.2 模拟对话的架构影响

模拟对话是基于案例的**衍生应用**：

```
真实案例（原始数据）
    ↓
模拟对话场景1（基于案例背景，练习技术A）
模拟对话场景2（基于案例背景，练习技术B）
模拟对话场景3（基于案例背景，换一种策略）
```

**关键特征**：
- 一个案例可以衍生多个模拟对话
- 模拟对话数据独立存储，不污染案例本身
- 需要双向引用：案例→模拟对话列表，模拟对话→源案例

**数据结构预留**：
```json
{
  "case_id": "C20260620001",
  "simulation_sessions": [              // 新增字段
    {
      "session_id": "sim_001",
      "created_at": "2026-07-01",
      "title": "练习共情技术",
      "dialogue_file": "D:/.../.../C20260620001_sim_001.json"
    }
  ]
}
```

## 三、架构改进方案

### 3.1 核心原则

1. **保持单向数据流的纯净性**：Word → JSON 不变
2. **在JSON上添加可编辑层**：基础层（编辑） + 学习层（新增）
3. **HTML从展示层升级为交互层**：支持编辑 → 写回JSON → 重新生成
4. **索引自动维护**：编辑后自动更新索引

### 3.2 新的数据流

```
Word（不可变）
    ↓ [AI处理] 
JSON案例（可编辑）
    ↓ [生成HTML]
HTML案例库（交互式）
    ↓ [用户编辑]
pending_updates.json（中转）
    ↓ [Python处理]
更新JSON + 更新索引 + 重新生成HTML
    ↓
用户刷新浏览器，看到最新内容
```

### 3.3 数据结构完善

**案例JSON新增字段**：

```json
{
  // ====== 不可变层 ======
  "case_id": "C20260620001",
  "source_file": "...",
  "created_at": "2026-06-20T21:51:04",
  "version": "1.0",                     // 新增：数据版本
  
  // ====== 基础层（可编辑）======
  "basic_info": {...},
  "session_info": {...},
  "tags": {...},
  "crisis_level": "L",
  "keywords": [...],
  "techniques_used": [...],
  "dialogue": "...",
  "ai_analysis": {...},
  
  // ====== 元数据层 ======
  "last_modified": "2026-06-27T15:30:00",  // 新增：最后修改时间
  "modification_history": [                // 新增：修改历史（可选）
    {
      "timestamp": "2026-06-27T15:30:00",
      "field": "tags.relation",
      "action": "added",
      "value": "家庭关系-亲子关系-情感忽视"
    }
  ],
  
  // ====== 学习层（新增）======
  "supervision_records": [              // 历次感悟
    {
      "id": "sr_20260627153000",
      "created_at": "2026-06-27T15:30:00",
      "updated_at": "2026-06-27T15:30:00",
      "content": "第一次阅读的感悟..."
    }
  ],
  
  // ====== 扩展层（预留）======
  "simulation_sessions": [],            // 模拟对话记录
  "related_cases": [],                  // 相关案例
  "learning_notes": []                  // 学习笔记（未来）
}
```

### 3.4 supervision/ 目录的重新定位

**当前问题**：
- JSON中有 `supervision_records` 数组
- 目录中有 `supervision/` 文件夹
- 两者功能重叠，容易混淆

**建议方案**：

**方案A（推荐）**：废弃 `supervision/` 目录，全部用JSON存储
- 优点：数据集中，易于维护
- 缺点：感悟内容长时JSON会很大

**方案B**：混合存储
- `supervision_records` 只存元数据（id、时间、标题）
- 内容存储在 `supervision/C20260620001_sr_001.md`
- 优点：JSON轻量，内容可独立编辑
- 缺点：数据分散，维护复杂

**我倾向方案A**，理由：
1. 案例数量不大（个人使用，预计几百个）
2. 感悟通常不超过1000字
3. 数据集中更易备份和迁移

### 3.5 索引自动维护机制

**当前问题**：
- 4个索引文件（relation_tags, symptom_tags, crisis_levels, keywords）
- 修改案例JSON后，索引不会自动更新
- 容易出现数据不一致

**解决方案**：

1. **编辑时触发索引更新**：
```python
def update_case(case_id, field, value):
    # 1. 更新案例JSON
    case_data = load_case(case_id)
    case_data[field] = value
    save_case(case_id, case_data)
    
    # 2. 自动更新索引
    rebuild_indexes()  # 重新扫描所有案例，更新索引
    
    # 3. 重新生成HTML
    generate_case_library()
```

2. **增量更新索引**（优化版）：
```python
def update_case_tags(case_id, old_tags, new_tags):
    # 1. 更新案例JSON
    # ...
    
    # 2. 增量更新索引
    removed_tags = set(old_tags) - set(new_tags)
    added_tags = set(new_tags) - set(old_tags)
    
    for tag in removed_tags:
        index_remove_case(tag, case_id)
    
    for tag in added_tags:
        index_add_case(tag, case_id)
```

## 四、实施方案对比

### 4.1 方案A：轻量级改造（推荐优先）

**思路**：在现有架构上最小化改动，快速实现核心功能

**实施步骤**：
1. ✅ 历次感悟：直接在JSON的 `supervision_records` 中存储
2. ✅ 标签管理：HTML编辑模式 → 写入 `pending_updates.json` → Python处理
3. ✅ 案例编辑：同上，支持常用字段编辑
4. ✅ 索引维护：编辑后全量重建索引（简单可靠）

**优点**：
- 改动小，风险低
- 快速验证需求
- 保持现有架构的简洁性

**缺点**：
- 编辑体验略差（需要手动运行Python脚本）
- 索引全量重建性能较差（但案例少时无影响）

### 4.2 方案B：架构升级（未来考虑）

**思路**：引入本地服务器，实现真正的前后端分离

**技术栈**：
- 后端：Flask/FastAPI 本地API服务
- 前端：HTML + JavaScript（Ajax请求）
- 数据库：SQLite（可选，或继续用JSON）

**优点**：
- 用户体验好（实时保存）
- 支持复杂功能（批量操作、撤销重做）
- 扩展性强

**缺点**：
- 需要重构现有代码
- 引入服务器复杂度
- 学习成本高

**建议**：先实施方案A，验证需求后再考虑方案B

## 五、关键设计决策

### 5.1 编辑粒度

**需要支持编辑的字段**：
- ✅ 标签（relation/symptom）
- ✅ 危机等级 + 证据
- ✅ 关键词
- ✅ 使用技术
- ✅ AI分析内容（摘要、优势、改进建议）
- ⚠️ 对话记录（慎重，可能很长）
- ⚠️ 基本信息（影响统计，需要同步更新索引）

**不应该编辑的字段**：
- ❌ case_id（唯一标识）
- ❌ source_file（溯源）
- ❌ created_at（时间戳）

### 5.2 版本控制策略

**简化方案（推荐）**：
- 只记录最后修改时间（`last_modified`）
- 不保留修改历史（如需回滚，从备份恢复）

**完整方案**（可选）：
- 记录每次修改的详细历史（`modification_history`）
- 支持查看历史版本
- 支持回滚到任意版本

**建议**：先用简化方案，未来有需要再升级

### 5.3 数据安全

**备份策略**：
1. 编辑前自动备份JSON到 `data/backup/`
2. 按日期组织：`data/backup/20260627/C20260620001.json`
3. 定期清理旧备份（保留最近30天）

**错误恢复**：
1. 如果编辑出错，从备份恢复
2. 提供 `restore.py` 脚本快速恢复

## 八、风险与应对

### 风险1：数据不一致
**场景**：修改标签后，索引未更新，搜索不到案例  
**应对**：编辑后自动触发索引重建

### 风险2：编辑冲突
**场景**：同时编辑同一案例  
**应对**：单用户系统，不考虑并发

### 风险3：数据丢失
**场景**：编辑失败，数据损坏  
**应对**：编辑前自动备份，提供恢复脚本

### 风险4：性能问题
**场景**：案例增多后，全量重建索引变慢  
**应对**：先不优化，有问题再切换到增量更新

## 九、总结与建议

### 核心观点

1. **这是一个学习系统，不只是展示系统**
   - 需要反馈回路（编辑 → 更新 → 重新展示）
   - 需要学习层（感悟、笔记、模拟对话）

2. **当前架构的本质是"单向数据流"**
   - Word → JSON → HTML
   - 缺少回流机制

3. **改进方向：在保持简洁的前提下添加编辑能力**
   - 不引入复杂架构（数据库、服务器）
   - 用简单的中转文件（pending_updates.json）
   - 保持JSON为唯一数据源

4. **分阶段实施，避免过度设计**
   - Phase 1：核心编辑功能（感悟+标签）
   - Phase 2：扩展编辑功能（更多字段）
   - Phase 3：高级功能（模拟对话）

### 立即行动

**建议优先实施**：
1. 历次感悟功能（最小可用版本）
2. 标签管理功能（最小可用版本）
3. 验证完整流程（编辑 → 更新 → 展示）
4. 收集用户反馈，迭代优化

**暂缓实施**：
1. 案例内容全字段编辑（先做常用字段）
2. 修改历史查看（先用备份恢复）
3. 模拟对话（需求不紧急，预留接口）
