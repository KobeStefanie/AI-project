# Context压缩前后的对话整理规则

## 核心原则

**在context即将压缩(compact)之前，主动整理当前对话并保存，压缩后立即读取恢复上下文。**

## 何时触发

- Token使用接近上限（如超过180k/200k）
- 系统提示即将进行context compaction
- 长时间连续工作后准备进入新阶段

## 压缩前操作流程

### 1. 创建压缩对话实录

在项目根目录创建/使用 `.claude/压缩对话实录/` 文件夹，按时间戳命名文件：

```
.claude/压缩对话实录/
  └── 2026-06-29_16-30_问题修复记录.md
  └── 2026-06-29_18-00_功能开发记录.md
```

### 2. 整理内容结构

每个压缩对话实录文件应包含：

```markdown
# 对话实录 - [主题]

**时间**：2026-06-29 16:30-18:00
**Token使用**：已用 180k/200k
**主要工作**：[简要描述]

## 关键上下文

### 当前状态
- 正在进行的任务
- 已完成的步骤
- 待办事项

### 重要决策
- 技术选型决策
- 架构调整
- 问题解决方案

### 关键文件
- [文件路径] - 修改内容说明
- [文件路径] - 新增功能说明

### 遗留问题
- [ ] 待解决的问题
- [ ] 需要验证的功能
- [ ] 后续优化点

## 详细记录

### [工作项1]
- 问题描述
- 解决方案
- 相关代码/文件

### [工作项2]
...

## 下一步计划

1. 压缩后首先要做的事情
2. 需要继续的工作
3. 需要验证的内容
```

### 3. 保存提示

告知用户："Context即将压缩，我已将当前对话整理保存到 `.claude/压缩对话实录/[文件名].md`，压缩后会自动读取恢复。"

## 压缩后操作流程

### 1. 立即读取最新实录

```bash
# 找到最新的压缩对话实录
ls -lt .claude/压缩对话实录/ | head -2
```

### 2. 向用户确认

"Context已压缩，正在读取压缩前的对话实录恢复上下文..."

### 3. 读取并总结

读取最新的实录文件，快速总结：
- 之前在做什么
- 当前状态如何
- 接下来要做什么

### 4. 继续工作

基于恢复的上下文，无缝继续之前的工作。

## **Why**: 为什么需要这个规则

1. **上下文连续性**：避免压缩后丢失重要的工作上下文
2. **决策追溯**：保留关键的技术决策和原因
3. **效率提升**：压缩后快速恢复，不需要用户重新解释
4. **知识积累**：形成项目的历史记录和知识库

## **How to apply**: 具体应用

### 项目结构

每个项目都应有：
```
项目根目录/
  └── .claude/
      ├── 压缩对话实录/
      │   ├── 2026-06-29_16-30_问题修复.md
      │   └── 2026-06-29_18-00_功能开发.md
      └── [其他claude配置]
```

### 命名规范

- 格式：`YYYY-MM-DD_HH-MM_主题描述.md`
- 主题描述应简洁明确（如：流派配置修复、数据结构迁移）
- 中文命名，便于快速识别

### 读取顺序

压缩后按此顺序恢复上下文：
1. 读取最新的压缩对话实录
2. 检查"当前状态"和"下一步计划"
3. 向用户确认是否继续之前的工作
4. 开始执行

### 特殊情况

- **多次压缩**：每次压缩都创建新文件，保留历史
- **跨天工作**：新一天开始时，总结前一天的工作
- **项目切换**：切换项目时，各自读取对应项目的实录

## 示例场景

### 场景1：功能开发中压缩

```
[压缩前 - 16:30]
我：正在实现流派配置的自动联动功能，已完成保存逻辑，准备测试

[压缩后 - 16:35]
Claude读取实录：之前在实现流派配置自动联动，保存逻辑已完成。
现在继续：开始测试自动联动功能。
```

### 场景2：问题修复中压缩

```
[压缩前]
正在修复JavaScript函数加载顺序问题，已定位原因是HTML生成顺序错误

[压缩后]
Claude读取实录：正在修复JS加载顺序问题，原因是HTML生成顺序。
接下来：继续重构generate_visit_details.py的模板结构。
```
