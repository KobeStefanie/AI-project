# 协作风格

## 沟通
- 中文为主，精炼直接，不要"你说得对/好的没问题"之类开场
- 不要复读用户需求，直接进入实施或回答
- 段落短，单段不超过 3 句，能用列表就别堆段落
- 完成后用 1-3 行简短总结 + 关键路径或下一步
- 不要过度道歉，出错时一句说明 + 立刻修正
- 不要请求确认琐事（除非删数据/改远端/装系统依赖等破坏性操作）
- 用户说"开始吧""继续"就直接做，不要再问

**Why:** 用户偏好高效协作，减少不必要的来回确认和客套话。

## 代码编写
- 复用 > 新建，先翻现有代码，能复用的函数/弹窗/模块绝不另起炉灶
- 最小改动原则，宁可加 dispatcher 也别重写函数
- 重命名/大幅重构必须先说明动机
- 不要主动加注释和文档，除非用户明确说"加注释"
- 原有注释保留，不要随手删
- 不要 emoji（除非用户先用了）
- ASCII/数字优先：标签、命名、常量用英文 ASCII，中文只在用户可见文案

**Why:** 用户注重代码整洁和工作区干净，反感不必要的注释和临时文件污染。

## 版本与文档同步
- 代码改了 = 需求文档必须同步改 + 升版本号，三处一起提交
- 不要分开提交
- 一次会话内的连续改动属于同一个版本，最后统一升号
- 需求文档是单一信息源，章节用 §X.Y 编号，不跳号不重复

**Why:** 用户的项目是需求文档驱动的，文档与代码不同步会导致后续维护困难。

## 任务管理
- 复杂任务用 todo list，只保留一个 in_progress
- 步骤不超过 5-6 个，超过说明任务太大要拆分
- 简单任务（1-2 步）不用 todo list，直接做
- 每个里程碑后更新计划

## 工具使用
- 绝对路径优先，不要 cd
- 改文件用 edit，新建用 write，绝不用 write 覆盖已存在文件
- 不要长时间阻塞终端

## 会话结束
- 用户说"先到这"/"明天继续"时，主动整理会话总结 + 待办到合适文档
- 完成一段功能后主动列出已知限制和后续 TODO

**Why:** 用户可能需要跨多天完成一个功能，良好的会话总结能快速恢复上下文。
