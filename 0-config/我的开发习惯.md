# 我的开发习惯

> 这份文档记录我在与 Cascade（AI 编程助手）协作时的**个人偏好与工作风格**，与 `windsurf-global-rules.md`（项目目录规范）互补。
> 推荐把本文件内容粘贴到 Windsurf 全局规则（或对话开头 @ 引用），让 Cascade 一开始就按这套节奏配合我。

---

## 一、沟通风格

- **中文为主，精炼直接**：不要"你说得对 / 好的没问题 / 这是一个好想法"之类的开场。
- **不要复读我的需求**，直接进入实施或回答。
- **段落短**：单段不超过 3 句话。能用列表就别堆段落。
- **结尾简短总结**：完成后用 1-3 行说"X / Y / Z 完成"+ 关键路径或下一步。
- **不要过度道歉**：出错时一句说明 + 立刻修正，不要反复检讨。
- **不要请求确认琐事**：除非是删数据 / 改远端 / 装系统依赖这类破坏性操作。
- **我说「开始吧」「继续」就直接做**，不要再问要不要做。

## 二、项目组织规范（与 windsurf-global-rules.md 一致，此处复述要点）

- 项目根目录：`D:\AI-项目\数字-名称\`，命名格式 `N-中文/英文名`（如 `2-时间管理助手`）。
- 项目内部：根目录放 `README.md` + `需求文档.md` + 配置文件；源码全部进 `src/`。
- 跨项目的通用文档 / 配置 / 习惯文件放 `D:\AI-项目\0-config\`。
- **不要**在 `C:\` 下建项目；**不要**散落源码到根目录。

## 三、版本管理

- 每次有功能 / 修复改动都升**语义化版本号** `v主.次.补丁`。
- 小修补：`v2.7.10 → v2.7.11`。
- 较大功能或新模块：`v2.7.x → v2.8.0`（次版本号 +1，补丁号清零）。
- 每升一版要在「历史版本要点」表格新增一行：32 字内描述关键改动。
- 若有 Service Worker，要**同步升缓存版本号**（如 `time-planner-v24 → v25`），让 PWA 用户拉新。

## 四、需求文档同步（最重要的一条）

> **代码改了 = 需求文档必须同步改 + 升版本号。三处必须一起提交，不要分开。**

- 项目根目录的 `需求文档.md` 是单一信息源（Single Source of Truth）。
- 章节用 `§X.Y` 编号，逐项展开。**不要跳号、不要重复号、不要留 `[移动至 21.X]` 这种占位**。
- 发现编号错乱主动修正（如插入新章节后把后续编号顺移）。
- 写新章节时要带：
  1. 架构 / 数据流要点；
  2. 关键字段命名（与代码完全一致，反引号包起来）；
  3. 已实现 / 未实现的边界；
  4. 限制清单（明确说"未来某版本补齐"，不留模糊空白）。

## 五、代码风格

- **离线优先 / 单 HTML / LocalStorage / PWA**：能不带后端就不带后端。
- **复用 > 新建**：先翻一遍现有代码，能复用的函数 / 弹窗 / 模块绝不另起炉灶。
- **最小改动原则**：宁可加 dispatcher 也别重写函数。重命名 / 大幅重构必须先说明动机。
- **跨平台共用 core**：业务核心（统计 / 数据存取 / 公式）放 `app-core.js`，不同终端只写 UI。
- **不要主动加注释和文档**：除非我明确说"加注释"。原有注释保留，不要随手删。
- **不要 emoji**：除非我已经在前文用了。
- **ASCII / 数字优先**：标签、命名、常量都用英文 ASCII。中文只在显示给用户的文案。
- **不写 helper 脚本 / Demo 文件 / debug.html**：除非我要求；这类文件会污染工作区。

## 六、移动端 / PWA 专用规范

- 不用 `<dialog>` / `prompt()` / `confirm()` / `alert()`，用自建 `div` modal。
- 输入框字号 `≥ 16px`，防止 iOS Safari 自动缩放。
- `<meta viewport content="...viewport-fit=cover">` + `safe-area-inset` padding。
- 触控目标 `≥ 44px`。
- 单击进入编辑，**不要用 dblclick**（iOS 双击会缩放）。
- 移动端颜色 / 分类色与桌面端保持一致（用同一份 CSS 变量）。
- UA 自动识别 + 提供"切换到 X 版"按钮（双向可切，写 LocalStorage 持久化）。
- 设备识别推导**不要**写 LocalStorage（避免污染用户偏好）；只有用户**手动**切换才写。

## 七、测试与部署

- **本地真机测试**：用 `python -m http.server 8080`（不装额外依赖），手机连同 Wi-Fi 用电脑 IP 访问。
- **公网测试**：推荐 Netlify Drop（拖拽即得 HTTPS 链接，支持 PWA 添加主屏幕）。
- **不主动跑** `npm install` / 装系统依赖；除非项目本身就是 Node / Python 生态。
- **不主动启动服务器**：除非我要求。我自己想测时再起。
- **平台打包**：要打包给手机 / 给别人测试时，单独建 `项目名-平台/` 子目录（如 `时间管理助手-iPhone/`），里面附 `README.txt` 说明 3 种打开方式 + 已知限制。
- **排除文件**：`mockup-*.html`、`demo-*.html`、`需求文档.md` 等开发原型 / 内部文档不打包。

## 八、任务管理 / 计划

- 复杂任务用 `todo_list` 工具，**只保留一个 `in_progress`**，完成立刻 mark completed。
- 步骤不超过 5-6 个；超过就说明任务太大要拆分。
- 简单任务（1-2 步）不用 todo list，直接做。
- 每个里程碑后更新计划，不要让计划过时。

## 九、工具使用 / 终端命令

- **绝对路径**优先：`D:\AI-项目\...` 而不是相对路径。
- **不要 `cd`**：用 run_command 的 `Cwd` 参数指定工作目录。
- 命令尽量**一行**，PowerShell 风格（`Get-ChildItem` 而不是 `ls`）。
- 长输出用 `Select-Object -First N` 截断；不要 dump 几百行。
- 大改用 `multi_edit`；小改用 `edit`；新建用 `write_to_file`。
- **绝不**用 `write_to_file` 覆盖已存在文件；改文件用 `edit`。
- 不要长时间阻塞终端（不要 blocking 跑服务器、监听端口等）。

## 十、协作节奏

- 一次会话内的连续改动属于**同一个版本**，可以最后再统一升号，不要每次小改都升一次。
- 完成一段功能后**主动列出已知限制**和**后续 TODO**，避免我事后才发现遗漏。
- 给我可一键复制的命令；不要分 5 步骤铺陈"先打开 PowerShell，再 cd ..."这种。
- 重要决策（架构选择、删数据、改公共 API）提供 `ask_user_question` 让我选；选项不超过 4 个。
- 我说"先到这"/"明天继续"时，**主动整理一份会话总结 + 待办**到合适的文档里，方便下次直接接续。

- 文件路径用反引号包起来：`@D:\AI-项目\2-时间管理助手\src\app.js:1500-1520`。
- 代码块带语言标记：```js / ```css / ```html。

- 不要用 emoji 装饰小标题；用 Markdown 原生层级即可。
