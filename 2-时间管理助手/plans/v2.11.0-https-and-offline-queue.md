## 余下工作分 5 块

### 块 0:升级证书架构到 CA + leaf 两层(新增,前置)

**目标**:iPhone 装 CA 一次,永久有效;以后 IP 变 / SAN 变 / 同一 CA 下别的项目,iPhone 完全不感知。

**0.1** `tools/gen-cert/` 重构。把 `selfsigned` 包换成直接使用 `node-forge`(`selfsigned` 内部就是它,只是不暴露 CA-leaf API)。

**0.2** 拆成三个脚本:

- `gen-ca.js`:生成 CA 根证书 + 私钥;**幂等**——如果 `certs/ca-cert.pem` 已存在则跳过(避免误覆盖导致 iPhone 已装信任失效)
- `gen-leaf.js`:读取 CA → 生成签名 leaf 证书(含当前所有 LAN IP/mDNS SAN);**每次覆盖**
- `gen-all.js`:依次跑 0.1 → 0.2(首次安装用)

**0.3** 输出文件结构:

```
certs/
├── ca-cert.pem          # CA 证书 PEM(iPhone 用)
├── ca-cert.crt          # 与 ca-cert.pem 同内容,扩展名便于 iPhone Safari 识别
├── ca-key.pem           # CA 私钥(永久保密,泄漏需重做 + iPhone 重装)
├── leaf-cert.pem        # server 用证书
├── leaf-cert-chain.pem  # leaf + ca 拼接(给 https.createServer 用)
└── leaf-key.pem         # server 用私钥
```

**0.4** 改 `src/server.js` 与 `sync-server.js` 读 `leaf-cert-chain.pem` + `leaf-key.pem`(替代原 `cert.pem`/`key.pem`)。

**0.5** CA 属性:
- CN: `Time Planner Personal CA`
- O:  `Time Planner Personal`
- 有效期 10 年
- `basicConstraints: CA=true, pathlen=0`(只能签 1 层 leaf)
- `keyUsage: keyCertSign + cRLSign + digitalSignature`

**0.6** leaf 属性:
- CN: `<hostname>`(如 `DESKTOP-QRG0JNN`)
- O:  `Time Planner Local`
- 有效期 2 年(短一些没关系,反正脚本一行重生成)
- `basicConstraints: CA=false`
- `extKeyUsage: serverAuth + clientAuth`
- `subjectAltName`:所有 LAN IP + `<hostname>` + `<hostname>.local` + `localhost` + `127.0.0.1`

**0.7** 重启服务器后所有 HTTPS 验证用新链。

### 块 1:让 iPhone 拿到 CA 并信任

**1.1** `src/server.js` 增加 `/cert.crt`、`/ca.crt`、`/cert.pem` 路由 → 都返回 `certs/ca-cert.crt` 内容(分发 CA,不是 leaf);Content-Type `application/x-x509-ca-cert`。

  - 通过 **HTTP 6371**(不需要先信任)在 iPhone Safari 输入 `http://192.168.31.153:6371/cert.crt` → Safari 弹"已下载描述文件"。
  - HTTPS 6443 同样路径也工作,适合已经 trust 后做更新。

**1.2** 输出可复制的 iPhone 操作指令:

  - Safari 访问 `http://192.168.31.153:6371/cert.crt` → "允许"
  - 设置 → 通用 → VPN与设备管理 → 已下载的描述文件 → "安装"(输入锁屏密码 → 安装确认 2 次)
  - 设置 → 通用 → 关于本机 → 证书信任设置 → 打开 `Time Planner Personal CA` 开关

**1.3** **可选** Windows 端 trust CA(开发体验,免本机浏览器警告):一行 PowerShell `Import-Certificate -FilePath certs\ca-cert.crt -CertStoreLocation Cert:\CurrentUser\Root`。

### 块 3:阶段 3.5 离线变更队列 + online 自动 flush

**3.1** `src/app-core.js` syncClient 内部增强:

  - 新 localStorage key `tm_sync_pending_weeks` → 数组,元素 `{year, week, queuedAt}`
  - `pushWeek` 失败时,把 `(year, week)` 加入队列(去重)
  - `pushWeek` 成功时,把 `(year, week)` 从队列移除
  - 新方法 `flushPendingQueue()`:遍历队列,对每个待推送周尝试 `pushWeek`;失败保留在队列;成功移除
  - 新方法 `getPendingCount()`:供 UI 显示

**3.2** `src/app.js` 启动钩子:

  - `setupSyncUI()` 末尾 → 一次性 `flushPendingQueue()`(覆盖"启动时就在 WiFi 内"的情况)
  - 全局 `window.addEventListener('online', () => syncClient.flushPendingQueue())`
  - 同步面板新增一行小提示:"待推送 N 周"

**3.3** 状态点(右上角小圆点)新增视觉:

  - 当 `getPendingCount() > 0` 时,绿点改成"绿点 + 数字徽标"或"半透明蓝色叠加",提示用户有积压
  - 鼠标悬停文案:"有 N 周变更待同步,联网后自动推送"

## 一些已经替你决定的事

1. **不回退** v2.10.1(autoHost)与 v2.10.2(SW cache-first):它们对桌面端依然有用,且与 v2.11.0 兼容。
2. **不引入** 单独的"高级 → HTTPS 模式"开关:`autoHost=true` + `window.location.protocol` 已自动覆盖。
3. **不修改** `manifest.json` 的 `start_url`:保持相对路径 `./时间管理助手.html`,HTTP/HTTPS 都能用。
4. **离线队列无重试上限**:每次 `online` 事件触发 flush,失败的周自然保留下次再试,避免引入 backoff 计算误伤实时场景。
5. **不绑定 Windows trust CA**:这只是开发体验,Chrome 警告点"高级 → 继续"即可。
