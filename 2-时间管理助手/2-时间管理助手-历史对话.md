### 6. Hotfix：手机端 sync-dot 永不刷新 bug

#### 6.1 现象

- iPhone 手动点同步面板「立即推送」→ 日志显示「✓ 推送成功」
- 但状态点**一直蓝闪不变绿**
- 不动屏幕等 5 秒仍蓝

#### 6.2 根因（5 分钟定位）

- `updateSyncDot()` 用 `$('sync-dot')` getElementById
- 桌面端 HTML 里 `<span id="sync-dot">` 有 id
- 手机端 footer 是 `app.js` `renderM_Footer()` 渲染的 `<span class="sync-dot">`，**只有 class 没 id**
- 所以 `updateSyncDot` 永远只更新桌面那个，手机端 dot 停在首次渲染颜色（常是 pending 蓝）

#### 6.3 修复

```js
// updateSyncDot 改 querySelectorAll('.sync-dot')，同步更新所有 dot
var dots = document.querySelectorAll('.sync-dot');
for (var i = 0; i < dots.length; i++) {
  dots[i].className = 'sync-dot ' + cls;
  dots[i].title = title;
}
```

- SW v39 → v40 强制刷新让 iPhone 拿到新 app.js
- 用户实测：**变绿了 ✅**

## v2.11.0 关键技术决策汇总

| 决策点 | 选择 | 理由 |
|---|---|---|
| 证书架构 | CA + leaf 双层 | leaf 重生不影响 iPhone 信任；CA 装一次永久 |
| 证书工具 | `node-forge` 纯 JS | 不依赖 mkcert / openssl 二进制 / GitHub 网络 |
| iOS 18 PWA 限制应对 | 接受书签 + SW（不上公网） | 功能 95% 等同；公网部署不在 v2.11.0 ~ v2.14 目标 |
| 离线队列粒度 | 「周」级 | 简单可靠；服务端 LWW 合并 cell 级；带宽不是瓶颈 |
| WebSocket 推送 | 推迟到 v2.12 | HTTPS 上线后必须 wss://，与 v2.11.0 一起做风险大 |
| SW 缓存 install | filter 掉 cross-origin URL | 避免 CDN 不可达拖死整个 SW（v36 在某些网络下根因） |
| SW fetch 拦截 | 跳过 manifest/icon | 让 Safari 装 PWA 时拿原始网络响应（即便最终用户场景没救回 PWA，逻辑上正确） |

## v2.11.1 Bug 修复（2026-05-28，约 1 小时）

> **目标**：排查并修复 v2.11.0 CHANGELOG §22.10 / §22.11 两个待排查 bug。
> **范围**：pull 全量替换数据丢失 + 同步状态假阳性，仅改 app-core.js / app.js / SW。

### 1. §22.11 排查：iPhone 离线编辑内容丢失

**代码追踪路径**：

1. `commitCurrentCell()` → `saveCells()` → `localStorage.setItem()` ✅ 同步写入
2. `_emitSaveChange` → `_handleSaveChange` → `_schedulePush`（debounce 1.5s）
3. 1.5s 后 `pushWeek` → `_enqueuePending` → POST

离线时 localStorage 已保存，queue 也可能已入队。回到在线后：

4. 页面加载 → `syncClient.init()` → autoPull 800ms
5. `pullWeek` → `_applyServerWeekToLocal` → **从空对象起手**替换所有 cells/keyitems
6. 服务器返回旧数据（离线编辑尚未推上去）→ 本地新数据被覆盖 ❌

**根因确认**：`_applyServerWeekToLocal` 三段（cells/keyitems/review）均全量替换，无 `updatedAt` 比较。

**修复**：
- cells：`getCells()` 读现有 → 逐 cell 比较 `updatedAt` → serverTs >= localTs 才覆盖
- keyitems：同上
- review：比较 `updatedAt`，server >= local 才覆盖
- autoPull 延迟 800→2500ms，先 `flushOfflineQueue()` 再 `pullWeek()`

### 2. §22.10 排查：重启后同步状态假阳性

**代码追踪**：

1. `updateSyncDot` 基于 `s.status` 判定颜色（'connected'=绿 / 'error'=红 / 'disconnected'=灰）
2. `_state.status` 仅在 push/pull/autoPull/testConnection 时更新
3. 无周期性可达性验证 → 一旦变绿永不主动探测

**根因确认**：缺少心跳机制，状态为静态快照。

**修复**：
- 新增 `_startHealthCheck()`：30s 间隔 ping `/info`
- 仅在 `connected` / `error` 态执行
- 成功 → `connected`，失败 → `error` + `lastError`
- `init()` 启用同步时自启，`saveSyncConfig` 关闭时停止

### 3. 关键决策

| 决策点 | 选择 | 理由 |
|---|---|---|
| pull 合并策略 | 逐条 updatedAt 比较 | 保留本地离线编辑，避免全量替换 |
| 心跳间隔 | 30s | 足够及时发现服务器宕机/恢复，不造成显著网络开销 |
| 心跳覆盖范围 | 仅 connected/error | disconnected 态不主动探测（节省资源） |
| autoPull 时序 | 先 flush 后 pull | 确保离线编辑先推上去再拉，避免数据回退 |

## v2.11.2 修复（2026-05-28，约 2-3 小时连续会话）

> **目标**：修复 v2.11.1 心跳遗留缺口 + 用户确认深色模式/书签功能正常 + GFP 子类从 5 项扩展为 7 项。
> **范围**：心跳增强、SW 版本自检、工具栏刷新按钮、GFP 5→7 全链更新、需求文档同步。

### 1. 心跳 disconnected 缺口修复

**根因**：v2.11.1 心跳仅在 `connected`/`error` 态执行，`init()` 初始态为 `disconnected`，心跳跳过。若 autoPull 未执行（URL 为空等），状态永久卡在 `disconnected`。

**修复（`app-core.js`）**：
- 提取 `_healthCheckPing()` 独立函数，覆盖 `disconnected`/`connected`/`error` 三种状态
- 首次心跳 1s 后立即执行（`setTimeout`），后续每 30s（`setInterval`）
- `_stopHealthCheck` 同时清除两个定时器（`healthCheckTimer` + `healthCheckInitTimer`）

### 2. SW 更新自助化

**痛点**：修改 SW 版本号后，浏览器不会立即检测更新。旧 SW 缓存了旧 HTML/JS，新页面里的更新逻辑根本不会执行（鸡生蛋问题）。

**实施方案**：
- `app.js` 新增 `EXPECTED_CACHE_NAME` 常量 + `forceUpdateSW()` 函数（注销所有 SW → 重载）
- `registerServiceWorker` 注册后 3s 自检：若 caches 中无期望版本 → 自动调 `forceUpdateSW()`
- 工具栏新增 `🔄 刷新` 按钮（手动触发 `forceUpdateSW`）
- 创建 `sw-cleanup.html` 一次性清理页（首次手动用）
